# IEEE 830準拠ガイド

## 概要

IEEE 830-1998は、ソフトウェア要件仕様書（SRS: Software Requirements Specification）の国際標準です。明確で検証可能な要件を記述するためのフレームワークを提供します。

## IEEE 830の品質特性

### 1. 正確性（Correct）

- 各要件がユーザーの真のニーズを反映
- 誤解の余地がない明確な記述

### 2. 曖昧性のなさ（Unambiguous）

- 各要件が単一の解釈のみを許容
- 「適切に」「必要に応じて」等の曖昧表現を排除

### 3. 完全性（Complete）

- すべての入力に対して出力が定義
- 正常系・異常系・境界値が網羅

### 4. 一貫性（Consistent）

- 要件間に矛盾がない
- 用語が統一されている

### 5. 検証可能性（Verifiable）

- 各要件がテスト可能
- 成功/失敗の判定基準が明確

### 6. 修正可能性（Modifiable）

- 変更が容易な構造
- 重複記述がない（DRY原則）

### 7. 追跡可能性（Traceable）

- 要件の出所が明確
- 実装との対応関係が追跡可能

## SRSの標準構造

```
1. はじめに
   1.1 目的
   1.2 スコープ
   1.3 定義・略語
   1.4 参考資料

2. 全体説明
   2.1 製品の視点
   2.2 製品の機能
   2.3 ユーザー特性
   2.4 制約条件
   2.5 前提と依存関係

3. 詳細要件
   3.1 機能要件
   3.2 外部インターフェース要件
   3.3 性能要件
   3.4 論理データベース要件
   3.5 設計制約
   3.6 ソフトウェアシステム属性
```

## 要件記述のパターン

### 良い例（検証可能）

```
REQ-001: ログイン処理は、ユーザー名とパスワードの入力後、
200ms以内にレスポンスを返すこと。
```

### 悪い例（曖昧）

```
REQ-001: ログイン処理は高速に動作すること。
```

## 曖昧表現の排除リスト

| 避けるべき表現 | 改善例                          |
| :------------- | :------------------------------ |
| 適切に         | 「100文字以内で」「JSON形式で」 |
| 必要に応じて   | 具体的な条件を明記              |
| 可能な限り     | 数値目標を設定                  |
| など           | すべての項目を列挙              |
| 高速に         | 「200ms以内」                   |

## チェックリスト

- [ ] 各要件に一意のIDが付与されているか
- [ ] 曖昧な表現が残っていないか
- [ ] すべての要件がテスト可能か
- [ ] 要件間に矛盾がないか
- [ ] 前提条件が明示されているか
