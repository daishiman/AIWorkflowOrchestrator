# ベストプラクティスチェックリスト

## 設計フェーズ

### 単一責任原則
- [ ] コマンドは1つの明確な責任のみを持つ
- [ ] 説明に「〜と〜」が含まれない
- [ ] 独立して実行可能

### 組み合わせ可能性
- [ ] 他のコマンドと組み合わせ可能
- [ ] 標準的な入出力形式
- [ ] 副作用が予測可能

### 冪等性
- [ ] 複数回実行しても同じ結果
- [ ] 状態チェックを実装
- [ ] 不要な重複処理を回避

## 実装フェーズ

### 構造
- [ ] YAML Frontmatterが適切
- [ ] description が具体的
- [ ] argument-hint を提供（引数がある場合）

### セキュリティ
- [ ] allowed-tools で権限制限
- [ ] 破壊的操作は確認プロンプト
- [ ] 入力バリデーション実装

### エラーハンドリング
- [ ] 早期バリデーション
- [ ] 明確なエラーメッセージ
- [ ] ロールバック機能（破壊的操作）

### パフォーマンス
- [ ] 不要な処理を削減
- [ ] 並列化可能な処理を特定
- [ ] 適切なモデル選択（haikuでよいか？）

## テストフェーズ

- [ ] 正常系テスト
- [ ] 異常系テスト
- [ ] エッジケーステスト
- [ ] 冪等性テスト（2回実行）

## ドキュメントフェーズ

- [ ] 使用例を提供
- [ ] よくある問題を記載
- [ ] 前提条件を明記

## レビューフェーズ

- [ ] チーム規約に準拠
- [ ] 命名規則に従う
- [ ] 既存コマンドとの一貫性
