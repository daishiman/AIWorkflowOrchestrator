# 曖昧性パターンガイド

## 概要

このガイドは、5つの曖昧性パターンの詳細な検出手法と除去テクニックを提供します。
要件定義において曖昧な表現を体系的に特定し、具体的で測定可能な要件に変換するための実践的なリファレンスです。

## パターン別検出と除去の詳細

### パターン1: 量的曖昧性（Quantitative Ambiguity）

#### 検出ヒューリスティクス

**時間関連の曖昧性**:
```
検出キーワード:
- 「高速」「速い」「遅い」「迅速に」
- 「すぐに」「即座に」「リアルタイム」
- 「頻繁に」「たまに」「しばしば」
- 「定期的に」「随時」「適宜」

検出例:
✗ 「システムは高速に応答する」
✗ 「データは定期的に同期される」
✗ 「ユーザーはすぐに結果を確認できる」
```

**量・規模関連の曖昧性**:
```
検出キーワード:
- 「多い」「少ない」「大量」「大規模」
- 「大きい」「小さい」「軽い」「重い」
- 「複数」「いくつか」「数多く」
- 「十分な」「限定的な」

検出例:
✗ 「大量のデータを処理できる」
✗ 「多くのユーザーが同時にアクセス」
✗ 「十分なストレージ容量」
```

#### 除去テクニック

**1. 具体的な数値・単位の指定**:

```markdown
## パフォーマンス要件の明確化

Before（曖昧）:
「APIは高速に応答する」

After（明確）:
「APIは以下の応答時間要件を満たす:
- GET /api/users: 平均100ms、95パーセンタイルで200ms
- POST /api/orders: 平均300ms、95パーセンタイルで500ms
- GET /api/reports: 平均2s、95パーセンタイルで3s」

改善ポイント:
✓ 具体的な時間（ms、s）
✓ パーセンタイル指定（95パーセンタイル）
✓ エンドポイント別の基準
```

**2. 範囲と条件の明示**:

```markdown
## スループット要件の明確化

Before（曖昧）:
「大量のリクエストを処理できる」

After（明確）:
「システムは以下のスループット要件を満たす:
- 通常時: 1,000 req/sec を安定処理
- ピーク時: 5,000 req/sec を30分間継続処理可能
- 同時接続: WebSocket 10,000 接続をサポート
- データ量: 1秒あたり10,000レコードを処理」

改善ポイント:
✓ 通常時とピーク時の区別
✓ 継続可能時間の明示
✓ 同時接続数の具体化
✓ データ処理量の定量化
```

**3. 測定方法の明示**:

```markdown
## 測定可能な要件定義

Before（曖昧）:
「頻繁にバックアップを実行」

After（明確）:
「バックアップ要件:
- 頻度: 1時間ごとに増分バックアップ、毎日午前2時にフルバックアップ
- 保存期間: 増分バックアップ30日、フルバックアップ1年
- 検証: 週1回、自動リストアテストを実行
- 測定: CloudWatch Logsでバックアップ成功率を記録」

改善ポイント:
✓ 具体的な頻度（時間単位）
✓ バックアップ種別の明示
✓ 保存期間の定量化
✓ 検証方法と測定方法
```

### パターン5: 主体の曖昧性（Subject Ambiguity）

#### 検出ヒューリスティクス

```

- 「ユーザーは」（どのユーザー？）

- 「管理者は」（どの権限レベル？）

検出例:
✗ 「ユーザーはデータを削除できる」
✗ 「システムは自動的にバックアップする」
✗ 「管理者は設定を変更できる」
```

#### 除去テクニック

**1. アクターの具体化**:

```markdown
## 削除権限の明確化

Before（曖昧）:
「ユーザーはデータを削除できる」

After（明確）:
「データ削除権限:

**ロール別の権限**:
1. 一般ユーザー:
   - 自分が作成したデータのみ削除可能
   - 作成後24時間以内のみ削除可能
   - 論理削除（データは30日間保持）

   - 自組織のすべてのデータを削除可能

- いつ: タイムスタンプ（UTC）
- 何を: リソースID、リソースタイプ
- 保存先: 監査ログテーブル（削除後も永久保存）」

**2. コンポーネントとトリガーの明示**:

```markdown
## 自動バックアップの明確化

Before（曖昧）:
「システムは自動的にバックアップする」

After（明確）:
「自動バックアップ要件:

**実行主体とトリガー**:
- トリガー: AWS EventBridge（cron式: `0 2 * * *`）
- 実行: Lambda関数 `backup-database-function`
- 対象: PostgreSQL RDS インスタンス `prod-db-01`

**バックアップ処理フロー**:
1. EventBridgeが毎日午前2時（UTC）にLambda関数をトリガー
2. Lambda関数がRDSの手動スナップショットAPIを呼び出し
3. スナップショット名: `backup-{YYYY-MM-DD}-{HH-mm-ss}`
4. 完了通知をSlackチャンネル `#ops-alerts` に送信
5. 失敗時はPagerDutyでオンコール担当者にアラート

**バックアップ保存先**:
- RDSスナップショット: AWS アカウント内
- クロスリージョンコピー: us-east-1 → ap-northeast-1（災害対策）

**保存期間とクリーンアップ**:
- 日次バックアップ: 30日間保持
- 週次バックアップ（日曜日）: 90日間保持
- 月次バックアップ（月初）: 1年間保持
- クリーンアップ: Lambda関数 `cleanup-old-backups` が毎日実行

**リストア手順**:
- 手動実行: AWS コンソールまたはCLI
- 実行者: システム管理者のみ（IAM権限で制御）
- 所要時間: 約15分（データベースサイズ100GBの場合）」

改善ポイント:
✓ 「システム」を具体的なAWSサービスに特定
✓ トリガーの詳細（EventBridge cron式）
✓ 処理フローを5ステップで明示
✓ 保存、クリーンアップ、リストアの責任も定義
```

**3. 権限レベルと責任範囲の明確化**:

```markdown
## 設定変更権限の明確化

Before（曖昧）:
「管理者は設定を変更できる」

After（明確）:
「設定変更権限:

**権限レベル別の変更範囲**:

**1. システム管理者（最高権限）**:
- 変更可能な設定:
  - すべてのシステム設定（データベース接続、APIキー等）
  - ユーザーロールと権限の定義
  - 監査ログの保存期間
  - セキュリティポリシー（パスワード要件等）
- 変更方法:
  - 管理画面の「システム設定」メニュー
  - 変更前に2段階認証を要求
  - すべての変更は監査ログに記録

**2. 組織管理者（組織内のみ）**:
- 変更可能な設定:
  - 自組織のブランディング（ロゴ、配色）
  - 自組織のユーザー管理（追加、削除、ロール変更）
  - 自組織の通知設定
- 変更方法:
  - 管理画面の「組織設定」メニュー
  - パスワード再確認を要求
  - 変更履歴は組織内で閲覧可能

**3. 一般ユーザー（自分のプロフィールのみ）**:
- 変更可能な設定:
  - 自分のプロフィール情報（名前、メールアドレス）
  - パスワード
  - 通知の受信設定（ON/OFF）
- 変更方法:
  - 「プロフィール設定」画面
  - パスワード変更時は現在のパスワードを要求
  - メールアドレス変更時は確認メールを送信

**変更の承認フロー**:
- システム設定の変更: 変更後、別のシステム管理者がレビュー（4eyes principle）
- 組織設定の変更: 即座に反映、ロールバック可能
- プロフィール設定: 即座に反映

**変更の記録と監査**:
- 誰が: ユーザーID、ロール
- いつ: タイムスタンプ（UTC）
- 何を: 変更した設定項目、変更前後の値
- 保存先: 監査ログテーブル（永久保存）
- 閲覧権限: システム管理者のみ」

改善ポイント:
✓ 「管理者」を3つの権限レベルに分類
✓ 各レベルの変更範囲を完全に列挙
✓ 変更方法と承認フローを明示
✓ 監査ログの記録項目も定義
```

## 横断的な除去テクニック

### テクニック1: 5W1H分析

すべてのパターンに適用可能な質問フレームワーク:

```markdown
## 5W1H質問リスト

**Who（誰が）**:
- この操作を実行する主体は誰か？
- どの権限レベルのユーザーか？
- 複数のロールで権限が異なるか？

**What（何を）**:
- 具体的に何が行われるか？
- 対象となるデータ、リソースは何か？
- 処理の結果は何か？

**When（いつ）**:
- どのタイミングで実行されるか？
- トリガーとなるイベントは何か？
- 実行頻度はどの程度か？

**Where（どこで）**:
- どのコンポーネント、サービスで実行されるか？
- データの保存先はどこか？
- 実行環境（本番、ステージング等）は？

**Why（なぜ）**:
- この要件が必要な理由は何か？
- ビジネス価値は何か？
- 代替案はないか？

**How（どのように）**:
- 具体的な実装方法は？
- 測定方法は？
- 検証方法は？
```

### テクニック2: Given-When-Then形式への変換

曖昧な要件を受け入れ基準に変換:

```markdown
## Given-When-Then変換例

Before（曖昧）:
「ユーザーは適切にログインできる」

After（Given-When-Then）:

**シナリオ1: 正しい認証情報でのログイン**
```gherkin
Given ユーザーが登録済み（email: test@example.com, password: ValidPass123!）
When ログイン画面でメールアドレスとパスワードを入力
And 「ログイン」ボタンをクリック
Then ダッシュボード画面に遷移する
And セッショントークンがCookieに保存される（有効期限: 1時間）
```

**シナリオ2: 誤ったパスワード**
```gherkin
Given ユーザーが登録済み（email: test@example.com）
When ログイン画面で誤ったパスワードを入力
And 「ログイン」ボタンをクリック
Then エラーメッセージ「メールアドレスまたはパスワードが正しくありません」が表示される
And ログイン画面に留まる
And ログイン試行回数がインクリメントされる
And 5回失敗した場合、アカウントが15分間ロックされる
```

改善ポイント:
✓ 「適切に」を2つのシナリオに分解
✓ 前提条件（Given）を明確化
✓ アクション（When）を具体化
✓ 期待される結果（Then）を詳細に記述
```

### テクニック3: 境界値分析

曖昧な範囲を境界値で明確化:

```markdown
## 境界値分析例

Before（曖昧）:
「適切なファイルサイズ」

After（境界値分析）:

**ファイルサイズ要件**:

**境界値**:
- 最小値: 1バイト（空ファイルは拒否）
- 推奨値: 10MB以下
- 最大値: 50MB
- 超過時: エラーメッセージ「ファイルサイズは50MB以下にしてください」

**テストケース**:
| ケース | ファイルサイズ | 期待される動作 |
|--------|--------------|--------------|
| 最小境界-1 | 0バイト | ❌ エラー「空ファイルはアップロードできません」 |
| 最小境界 | 1バイト | ✅ アップロード成功 |
| 通常値 | 5MB | ✅ アップロード成功 |
| 推奨上限 | 10MB | ✅ アップロード成功 |
| 推奨超過 | 20MB | ⚠️ 警告「大きなファイルはアップロードに時間がかかります」 |
| 最大境界 | 50MB | ✅ アップロード成功 |
| 最大境界+1 | 50MB + 1バイト | ❌ エラー「ファイルサイズは50MB以下にしてください」 |

改善ポイント:
✓ 「適切な」を3つの境界値（最小/推奨/最大）で定義
✓ 境界値±1のテストケースを網羅
✓ 各ケースの期待動作を明示
```
