# リスク識別手法

## 概要

リスク識別は、プロジェクトに影響を与える可能性のある潜在的なリスクを
体系的に発見し、文書化するプロセスです。

## ブレインストーミング手法

### カテゴリー別リスク探索

**技術リスク**:

- 新技術・未経験技術の採用リスク
- 技術的実現可能性の不確実性
- パフォーマンス要件を満たせない可能性
- 統合の複雑性
- スケーラビリティの問題
- セキュリティ脆弱性
- 技術的負債の蓄積

**プロジェクトリスク**:

- スケジュール遅延
- リソース不足（人員・予算・設備）
- スコープクリープ
- 依存関係の複雑化
- コミュニケーション不全
- 変更管理の失敗
- ステークホルダーの期待値ギャップ

**ビジネスリスク**:

- 市場変化・競合の台頭
- 規制・法令変更
- ビジネス要件の変更
- 予算削減
- 組織変更
- 顧客ニーズの変化

**外部リスク**:

- ベンダー・サプライヤーの問題
- インフラ障害
- 自然災害
- 経済環境の変化
- 政治的要因

### ファシリテーション技法

**準備**:

- 参加者の選定（多様な視点）
- 時間枠の設定（60-90分）
- 環境準備（ホワイトボード、付箋）
- カテゴリー提示（事前共有）

**実施**:

1. アイデア発散（20-30分）
   - 批判禁止
   - 量重視
   - 自由な発想
   - すべて記録

2. グループ化（15-20分）
   - 類似リスクをまとめる
   - カテゴリーに分類
   - 重複を統合

3. 優先順位付け（15-20分）
   - 影響度と確率を評価
   - トップリスクを特定
   - 即座の対応が必要なものを識別

4. 文書化（10-15分）
   - リスクレジスターに記録
   - オーナーを割り当て
   - 次のステップを決定

### オンラインツール活用

**Miro / Mural**:

- デジタルホワイトボード
- 付箋機能
- 投票機能
- テンプレート利用可能

**Google Jamboard**:

- シンプルな付箋ツール
- リアルタイム共同編集
- 無料で利用可能

**Microsoft Whiteboard**:

- Teams統合
- インク機能
- 画像挿入可能

## SWOT分析

### フレームワーク

```
Strengths（強み）       Weaknesses（弱み）
┌─────────────────┐   ┌─────────────────┐
│ 内部の強み       │   │ 内部の弱み       │
│ - チームスキル   │   │ - 経験不足       │
│ - 技術資産       │   │ - リソース制約   │
│ - ブランド       │   │ - プロセス未熟   │
└─────────────────┘   └─────────────────┘

Opportunities（機会）    Threats（脅威）
┌─────────────────┐   ┌─────────────────┐
│ 外部の機会       │   │ 外部の脅威       │
│ - 市場成長       │   │ - 競合          │
│ - 技術トレンド   │   │ - 規制変更       │
│ - パートナー     │   │ - 技術陳腐化     │
└─────────────────┘   └─────────────────┘
```

### 実施手順

1. **Strengths（強み）の特定**
   - 何が得意か？
   - 競合優位性は？
   - リソースの強みは？
   - 過去の成功要因は？

2. **Weaknesses（弱み）の特定**
   - 何が不足しているか？
   - 競合に劣る点は？
   - 改善すべき領域は？
   - 過去の失敗要因は？

3. **Opportunities（機会）の特定**
   - 市場のトレンドは？
   - 技術革新の機会は？
   - ビジネスチャンスは？
   - 強みを活かせる機会は？

4. **Threats（脅威）の特定**
   - 競合の動きは？
   - 市場の変化は？
   - 技術の変化は？
   - 弱みが露呈するリスクは？

### リスクへの変換

**Weaknesses → リスク（脅威）**:

- 弱み: チームにReact経験者が少ない
- リスク: React開発でパフォーマンス問題が発生する可能性

**Threats → リスク（脅威）**:

- 脅威: 競合が類似機能をリリース予定
- リスク: 市場投入が遅れ、競争力を失う可能性

**Strengths → 正のリスク（機会）**:

- 強み: 高いDevOpsスキル
- 機会: 自動化により開発速度を大幅に向上できる

**Opportunities → 正のリスク（機会）**:

- 機会: AI技術の普及
- 機会: AI機能追加により差別化できる

## チェックリスト手法

### 技術リスクチェックリスト

**技術選定**:

- [ ] 技術の成熟度は十分か？（本番実績あり）
- [ ] チームの習熟度は？（経験者が30%以上）
- [ ] コミュニティサポートは活発か？
- [ ] ライセンスは商用利用可能か？
- [ ] 長期サポートは保証されているか？
- [ ] セキュリティパッチは定期的か？

**パフォーマンス**:

- [ ] 負荷テストは実施済みか？
- [ ] スケーラビリティは検証済みか？
- [ ] ボトルネックは特定されているか？
- [ ] パフォーマンス要件は明確か？
- [ ] モニタリング計画はあるか？

**統合**:

- [ ] 外部システムとの接続は検証済みか？
- [ ] API仕様は確定しているか？
- [ ] データフォーマットは合意済みか？
- [ ] 障害時のフォールバックは設計済みか？
- [ ] 統合テスト計画はあるか？

**セキュリティ**:

- [ ] セキュリティレビューは実施済みか？
- [ ] 脆弱性スキャンは実施したか？
- [ ] 認証・認可の設計は適切か？
- [ ] データ暗号化は実装されているか？
- [ ] コンプライアンス要件は満たしているか？

**技術的負債**:

- [ ] 技術的負債の返済計画はあるか？
- [ ] コード品質の基準は定義されているか？
- [ ] リファクタリングの優先順位は明確か？
- [ ] 自動テストのカバレッジは十分か？

### プロジェクトリスクチェックリスト

**スケジュール**:

- [ ] スケジュールは現実的か？（バッファ含む）
- [ ] クリティカルパスは特定されているか？
- [ ] 依存関係は明確か？
- [ ] マイルストーンは適切か？
- [ ] 遅延時の対策は計画されているか？

**リソース**:

- [ ] 必要なスキルセットは揃っているか？
- [ ] 人員は確保されているか？
- [ ] 予算は十分か？
- [ ] ツール・環境は整っているか？
- [ ] リソース不足時の対策はあるか？

**スコープ**:

- [ ] 要件は明確か？
- [ ] スコープは合意済みか？
- [ ] 変更管理プロセスはあるか？
- [ ] 優先順位は定義されているか？
- [ ] スコープクリープ防止策はあるか？

**コミュニケーション**:

- [ ] コミュニケーション計画はあるか？
- [ ] レポートライン明確か？
- [ ] 会議体は適切か？
- [ ] 情報共有の仕組みはあるか？
- [ ] ステークホルダーマネジメント計画はあるか？

### カスタムチェックリスト作成

**ステップ1: 過去プロジェクト分析**

- 過去のリスクレジスター確認
- 発生頻度の高いリスクを特定
- 影響度の大きいリスクを特定

**ステップ2: 業界ベストプラクティス調査**

- 業界標準チェックリスト参照
- 専門家の意見聴取
- 最新トレンドの反映

**ステップ3: プロジェクト固有要因追加**

- ビジネスドメイン特有のリスク
- 組織文化的リスク
- 技術スタック固有のリスク

**ステップ4: 定期レビューと更新**

- 四半期ごとにレビュー
- 新規リスク追加
- 陳腐化した項目削除

## デルファイ法

### 概要

専門家の匿名意見を複数ラウンド収集し、合意を形成する手法。
リスク識別において、客観性と多様性を確保できる。

### 実施手順

**ラウンド1: 初期リスク収集**

1. 専門家に質問票送付
2. 匿名でリスクを列挙してもらう
3. すべての回答を収集・整理

**ラウンド2: フィードバックと再評価**

1. ラウンド1の結果を統合し、専門家に共有
2. 各リスクの確率と影響度を評価してもらう
3. 評価理由を記述してもらう

**ラウンド3: 合意形成**

1. ラウンド2の結果（統計値と理由）を共有
2. 再度評価を依頼
3. 合意が得られるまで繰り返す（通常2-4ラウンド）

### メリット

- 集団思考の回避
- 権威バイアスの排除
- 多様な視点の統合
- 客観的な評価

### デメリット

- 時間がかかる
- 専門家の選定が難しい
- ファシリテーションが必要

## チェックリスト分析（What-If分析）

### 手法

「もし～だったら？」という問いを体系的に投げかけ、
想定外のリスクを発見する。

### 質問例

**技術面**:

- もし主要ライブラリがサポート終了したら？
- もしAPIのレスポンスタイムが10倍になったら？
- もしデータベースが突然読み取り専用になったら？
- もしCDNが利用不可になったら？
- もし暗号化アルゴリズムが破られたら？

**プロジェクト面**:

- もしキーパーソンが退職したら？
- もし予算が50%削減されたら？
- もしスケジュールが半分になったら？
- もしステークホルダーが交代したら？
- もし要件が大幅に変更されたら？

**ビジネス面**:

- もし競合が先に市場投入したら？
- もし規制が厳格化されたら？
- もし顧客ニーズが変化したら？
- もし経済危機が発生したら？

### 実施のコツ

- 極端なシナリオも検討する
- 複合的な問題を想定する
- 過去の類似事例を参照する
- 発散思考を奨励する

## 前提条件分析

### 概要

プロジェクト計画の前提条件を明示化し、
その前提が崩れた場合のリスクを特定する。

### 前提条件の洗い出し

**技術的前提**:

- ライブラリXは継続的にサポートされる
- クラウドプロバイダーの可用性は99.9%以上
- データ移行は2週間で完了する
- パフォーマンステストは1週間で完了する

**プロジェクト前提**:

- チームメンバーは安定している
- 予算は確保されている
- ステークホルダーの優先順位は変わらない
- 他プロジェクトとの依存関係はない

**ビジネス前提**:

- 市場環境は安定している
- 規制は変更されない
- 競合状況は予測通り
- 顧客ニーズは変化しない

### リスク変換

各前提条件に対して:

1. 前提が崩れる確率を評価
2. 崩れた場合の影響を評価
3. 高リスクの前提をリスクレジスターに追加
4. 対策を検討

**例**:

- 前提: データ移行は2週間で完了する
- リスク: データ移行が遅延し、リリースが2週間遅れる
- 確率: 中（40%）
- 影響: 重大（リリース遅延）
- 対策: 段階的移行、パラレル実行、外部専門家投入

## ルートコーズ分析（事前）

### 5 Whys

過去の類似プロジェクトの問題に対して「なぜ？」を5回繰り返し、
根本原因を特定し、同様のリスクを予防する。

**例**:

1. なぜリリースが遅延したか？ → テストに時間がかかった
2. なぜテストに時間がかかったか？ → バグが多かった
3. なぜバグが多かったか？ → コードレビューが不十分
4. なぜコードレビューが不十分だったか？ → レビュアーが不足
5. なぜレビュアーが不足したか？ → レビュー計画がなかった

**リスク**: レビュー計画不足によるバグ多発と遅延
**対策**: レビュー計画の策定、レビュアーのアサイン

### フィッシュボーン（特性要因図）

問題の原因を体系的に分類し、可視化する。

```
人（Man）      方法（Method）
  \              /
   \            /
    \          /
     \        /
      \      /
       \    /
        問題
       /    \
      /      \
     /        \
    /          \
   /            \
  /              \
機械（Machine） 材料（Material）
```

カテゴリー別に原因をブレインストーミングし、
潜在的リスクを発見する。

## まとめ

リスク識別は継続的なプロセスです。
複数の手法を組み合わせることで、
より包括的にリスクを特定できます。

**推奨アプローチ**:

1. プロジェクト開始時: ブレインストーミング + SWOT
2. 計画フェーズ: チェックリスト + 前提条件分析
3. 実行フェーズ: What-If分析（定期的）
4. レビュー時: デルファイ法（必要に応じて）

**記録と管理**:

- すべてのリスクをリスクレジスターに記録
- 定期的にレビュー（週次/隔週）
- 新規リスクの追加
- クローズしたリスクの整理
