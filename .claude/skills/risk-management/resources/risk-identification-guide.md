# リスク識別ガイド

## 概要

このガイドは、プロジェクトにおける潜在的リスクを体系的に識別するための実践的手法を提供します。

## リスク識別の目的

- **早期発見**: 問題が顕在化する前にリスクを特定
- **包括的評価**: 技術、プロジェクト、ビジネス、外部の全領域をカバー
- **チーム参加**: 多様な視点からリスクを洗い出す
- **継続的改善**: 定期的にリスク状況を見直す

---

## 1. ブレインストーミング手法

### 実施タイミング

- プロジェクト開始時
- 主要マイルストーン前
- アーキテクチャ変更時
- インシデント発生後

### 準備

```yaml
participants:
  - プロダクトオーナー
  - テックリード
  - 開発チーム
  - QAエンジニア
  - インフラエンジニア
  - セキュリティ担当者

duration: 60-90分
materials:
  - ホワイトボード/付箋
  - リスクカテゴリーリスト
  - 過去のリスクレジスター
```

### プロセス

#### Step 1: カテゴリー別探索（30分）

```
技術リスク:
□ 新技術の採用リスク
  - 学習曲線、成熟度、コミュニティサポート
□ システム統合リスク
  - API互換性、データ移行、レガシーシステム連携
□ パフォーマンスリスク
  - スケーラビリティ、レスポンスタイム、負荷分散
□ セキュリティリスク
  - 脆弱性、認証/認可、データ保護
□ 技術的負債リスク
  - コード品質、保守性、将来拡張性

プロジェクトリスク:
□ スケジュールリスク
  - 見積もり精度、依存関係、バッファ不足
□ リソースリスク
  - 人員不足、スキルギャップ、離職
□ スコープリスク
  - スコープクリープ、要件変更、優先順位変動
□ コミュニケーションリスク
  - 情報共有不足、誤解、ステークホルダー調整
□ 変更管理リスク
  - プロセス未確立、レビュー遅延、承認プロセス

ビジネスリスク:
□ 市場リスク
  - 競合優位性、タイミング、市場変化
□ 規制リスク
  - 法規制変更、コンプライアンス要件
□ 予算リスク
  - コスト超過、ROI未達成
□ ユーザー受容リスク
  - UX問題、期待値ギャップ

外部リスク:
□ ベンダーリスク
  - サービス品質、SLA、依存度
□ インフラリスク
  - クラウド障害、ネットワーク、データセンター
□ 不可抗力リスク
  - 災害、パンデミック、その他外部要因
```

#### Step 2: グルーピングとマージ（15分）

```
類似リスクの統合:
1. 同じ根本原因を持つリスクをグループ化
2. 重複するリスクを統合
3. 包含関係を整理（親リスク・子リスク）

例:
- "APIレート制限" + "API障害" → "外部API依存リスク"
- "人員不足" + "スキル不足" → "リソース不足リスク"
```

#### Step 3: 優先順位付け（20分）

```
各リスクに対して簡易評価:
- 高/中/低の3段階で確率を評価
- 高/中/低の3段階で影響を評価
- 組み合わせで優先度を決定

優先マトリクス:
      影響度
    低   中   高
高 │中  │高  │最優先│
確 │    │    │      │
率 中 │低  │中  │高    │
   │    │    │      │
低 │低  │低  │中    │
```

#### Step 4: 記録とフォローアップ（10分）

```
記録内容:
- リスクID（R-001, R-002, ...）
- リスク名（簡潔な説明）
- カテゴリー
- 確率/影響度（暫定）
- 担当者（詳細分析を実施）
- 期限（次回レビュー日）
```

### ブレインストーミングのベストプラクティス

```
✅ 良い実践:
- 批判なし: どんな意見も歓迎
- 量を重視: 初期段階では多数のアイデアを出す
- 多様な視点: 異なる役割の人を招待
- 具体例使用: 過去の失敗から学ぶ
- タイムボックス: 時間を区切って集中

❌ 避けるべき実践:
- 即座の評価: ブレストと評価を混ぜない
- 支配的な発言: 全員が均等に参加
- 楽観的バイアス: "うちは大丈夫"思考
- 政治的配慮: リスクを隠さない
- 表面的分析: 根本原因を探る
```

## 2. SWOT分析

### 目的

組織やプロジェクトの内部環境（強み・弱み）と外部環境（機会・脅威）を分析し、戦略的リスクと機会を特定する。

### SWOT四象限

```
内部要因（コントロール可能）
┌─────────────────┬─────────────────┐
│  Strengths      │  Weaknesses     │
│  （強み）       │  （弱み）       │
│  - リソース     │  - 制約         │
│  - スキル       │  - スキルギャップ│
│  - 技術資産     │  - 技術的負債   │
│  - ブランド     │  - プロセス不備 │
└─────────────────┴─────────────────┘
外部要因（コントロール不可）
┌─────────────────┬─────────────────┐
│  Opportunities  │  Threats        │
│  （機会）       │  （脅威）       │
│  - 市場トレンド │  - 競合         │
│  - 新技術       │  - 規制変更     │
│  - パートナー   │  - 経済状況     │
│  - ユーザーニーズ│ - 技術陳腐化   │
└─────────────────┴─────────────────┘
```

### 実施手順

#### Step 1: 強み（Strengths）の特定

```
質問:
- 他社にない技術的優位性は？
- チームの専門スキルは？
- 既存の資産や顧客基盤は？
- 成功したプロジェクトの要因は？

例:
- 高度なマイクロサービス実装経験
- 強固なDevOpsパイプライン
- 既存顧客1000社のデータ
- クラウドネイティブ設計の知見
```

#### Step 2: 弱み（Weaknesses）の特定

```
質問:
- リソースや予算の制約は？
- スキルギャップや人員不足は？
- 技術的負債やレガシーシステムは？
- プロセスや体制の問題は？

例:
- フロントエンドエンジニア不足
- モノリシックなレガシーコード
- セキュリティ専門家の不在
- 自動化テストのカバレッジ低下
```

#### Step 3: 機会（Opportunities）の特定

```
質問:
- 市場やユーザーニーズの変化は？
- 新しい技術やツールのトレンドは？
- パートナーシップの可能性は？
- 規制緩和やインセンティブは？

例:
- AIによる業務自動化需要増加
- サーバーレス技術の成熟
- 大手企業とのAPI連携打診
- データプライバシー意識の高まり
```

#### Step 4: 脅威（Threats）の特定

```
質問:
- 競合の動向や新規参入は？
- 技術の陳腐化リスクは？
- 規制強化やコンプライアンスは？
- 経済状況や市場縮小は？

例:
- 大手テック企業の類似サービス参入
- React 18への移行圧力
- GDPR/個人情報保護法の強化
- IT予算削減トレンド
```

### SWOTからリスク/機会への変換

```yaml
# 強み → 機会活用戦略
strategy_1:
  strength: "マイクロサービス経験"
  opportunity: "クラウド需要増加"
  action: "SaaS製品ラインナップ拡充"

# 強み → 脅威防御戦略
strategy_2:
  strength: "強固なセキュリティ"
  threat: "規制強化"
  action: "コンプライアンス機能を差別化"

# 弱み → 機会逸失リスク
risk_1:
  weakness: "AI人材不足"
  opportunity: "AI需要増加"
  risk: "競合に市場シェアを奪われる"
  mitigation: "外部パートナーとの協業"

# 弱み → 脅威増幅リスク
risk_2:
  weakness: "レガシーコード"
  threat: "技術陳腐化"
  risk: "保守コスト増大・人材流出"
  mitigation: "段階的モダナイゼーション計画"
```

## 3. チェックリスト手法

### 技術面チェックリスト

```markdown
## 技術的実現可能性

- [ ] 技術スパイクやPoC実施済みか
- [ ] 技術的制約事項を理解しているか
- [ ] サードパーティライブラリの成熟度は十分か
- [ ] チームに必要な技術スキルが存在するか
- [ ] 技術選択の代替案を検討したか

## パフォーマンス

- [ ] 負荷テストシナリオを定義したか
- [ ] スケーリング戦略は明確か
- [ ] ボトルネックとなりうる箇所を特定したか
- [ ] パフォーマンス要件（SLA）が定義されているか
- [ ] モニタリング・アラート戦略はあるか

## スケーラビリティ

- [ ] ユーザー数増加時のアーキテクチャ対応は可能か
- [ ] データ量増加に耐えうるか
- [ ] 水平スケーリングが可能な設計か
- [ ] キャッシュ戦略は適切か
- [ ] データベース分散の必要性を評価したか

## セキュリティ

- [ ] 脅威モデリングを実施したか
- [ ] 認証/認可メカニズムは適切か
- [ ] データ暗号化（保存時・転送時）は実装されるか
- [ ] セキュリティテストは計画されているか
- [ ] 脆弱性スキャンのプロセスはあるか

## 技術的負債

- [ ] 既存コードの品質は受容可能か
- [ ] リファクタリングの優先順位は明確か
- [ ] テストカバレッジは十分か
- [ ] ドキュメントは保守されているか
- [ ] 技術的負債の返済計画はあるか
```

### プロジェクト面チェックリスト

```markdown
## スケジュール

- [ ] 見積もりは楽観的すぎないか
- [ ] バッファ（20-30%）を確保したか
- [ ] 依存関係によるブロッカーを特定したか
- [ ] クリティカルパスを明確にしたか
- [ ] マイルストーンは現実的か

## リソース

- [ ] 必要な人員は確保されているか
- [ ] スキルギャップへの対策はあるか
- [ ] 休暇・休職の影響を考慮したか
- [ ] 外部リソース（ベンダー）の契約は締結済みか
- [ ] バックアップ要員は用意されているか

## 依存関係

- [ ] 外部チームへの依存を明確にしたか
- [ ] サードパーティAPIの依存度を評価したか
- [ ] インフラ準備の依存関係を特定したか
- [ ] ブロッカーとなる承認プロセスはないか
- [ ] 依存関係の遅延時の代替策はあるか

## コミュニケーション

- [ ] ステークホルダーとの定例会議はセットされているか
- [ ] 報告フォーマットは合意されているか
- [ ] エスカレーションパスは明確か
- [ ] チーム内コミュニケーション手段は確立しているか
- [ ] タイムゾーンや言語の問題はないか

## 変更管理

- [ ] 変更要求プロセスは定義されているか
- [ ] スコープクリープの防止策はあるか
- [ ] 優先順位変更時の対応フローは明確か
- [ ] 変更の影響評価プロセスはあるか
- [ ] 変更承認の権限者は明確か
```

### ビジネス面チェックリスト

```markdown
## 市場・競合

- [ ] 競合分析を実施したか
- [ ] 市場のタイミングは適切か
- [ ] ユーザーニーズの検証はできているか
- [ ] 差別化要素は明確か
- [ ] 市場変化への対応策はあるか

## 規制・コンプライアンス

- [ ] 該当する規制（GDPR、個人情報保護法等）を特定したか
- [ ] コンプライアンスチェックリストを作成したか
- [ ] 法務レビューは完了しているか
- [ ] 監査対応の準備はあるか
- [ ] 規制変更の監視体制はあるか

## 予算・ROI

- [ ] 予算見積もりは正確か
- [ ] コスト超過時の対応策はあるか
- [ ] ROI計算は現実的か
- [ ] キャッシュフロー計画はあるか
- [ ] コスト削減の余地を検討したか

## ユーザー受容

- [ ] ユーザーテストを計画したか
- [ ] フィードバック収集の仕組みはあるか
- [ ] トレーニング/サポート体制はあるか
- [ ] リリース後のサポート計画はあるか
- [ ] ユーザーコミュニケーション戦略はあるか
```

### 外部面チェックリスト

```markdown
## ベンダー・パートナー

- [ ] ベンダーのSLAは明確か
- [ ] ベンダーロックインリスクを評価したか
- [ ] 契約条件は適切か
- [ ] エスカレーションプロセスは明確か
- [ ] 複数ベンダー戦略を検討したか

## インフラ・クラウド

- [ ] クラウドプロバイダーの信頼性を評価したか
- [ ] 障害時の復旧計画（DR）はあるか
- [ ] データバックアップ戦略は確立しているか
- [ ] ネットワーク障害への対策はあるか
- [ ] マルチリージョン対応は必要か

## 不可抗力

- [ ] 事業継続計画（BCP）はあるか
- [ ] リモートワーク体制は整っているか
- [ ] 主要メンバー不在時の対応策はあるか
- [ ] データセンター災害時の対策はあるか
- [ ] サプライチェーン中断への備えはあるか
```

## 4. 専門領域別リスクカテゴリー

### セキュリティリスク

```
- 認証/認可の脆弱性
- インジェクション攻撃（SQL、XSS、CSRF）
- データ漏洩・不正アクセス
- 暗号化不備
- APIセキュリティ
- サプライチェーン攻撃
- DDoS攻撃
```

### データリスク

```
- データ損失・破損
- データ不整合
- マイグレーション失敗
- バックアップ・リストア失敗
- データ品質問題
- GDPR/プライバシー違反
```

### 統合リスク

```
- API互換性問題
- データフォーマット不一致
- 統合ポイント障害
- バージョン非互換
- レガシーシステム依存
```

### クラウド/インフラリスク

```
- プロバイダー障害
- ネットワーク遅延・切断
- リソース制限超過
- コスト超過
- ベンダーロックイン
```

## 6. リスク識別のアンチパターン

```
❌ アンチパターン1: 楽観的バイアス
症状: "過去に問題なかったから大丈夫"
対策: データと事実に基づく評価

❌ アンチパターン2: グループシンク
症状: 反対意見が出ない、全員一致
対策: デビルズアドボケイト（批判役）を設定

❌ アンチパターン3: 表面的分析
症状: "リスクはない"という結論
対策: 5 Whysで根本原因を探る

❌ アンチパターン4: ブレーミング文化
症状: リスク報告を躊躇う雰囲気
対策: 心理的安全性の確保

❌ アンチパターン5: リスク識別の一度きり
症状: プロジェクト開始時のみ実施
対策: 定期的なリスクスキャン（週次/月次）
```
