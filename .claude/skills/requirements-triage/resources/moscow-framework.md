# MoSCoW分類フレームワーク

## MoSCoWとは

MoSCoW分類は、要件の優先順位を4つのカテゴリーに分類する手法です。
プロジェクトスコープを管理し、MVP定義を明確にするために使用します。

## 4つのカテゴリー

### Must have（必須）

**定義**: プロジェクトの成功に不可欠な要件

**判断基準**:

- システムの核心機能である
- これがなければプロジェクトは成立しない
- 法的義務、契約上の義務がある
- ビジネス価値が極めて高い（4-5点）

**実践例**:

```
ECサイトのMust have:
✓ 商品検索
✓ カート機能
✓ 決済機能（最低限の決済手段）
✓ 注文確認メール
✓ 在庫管理
```

### Should have（重要）

**定義**: 重要だが、必須ではない要件

**判断基準**:

- ビジネス価値は高い（3点以上）
- 短期的には回避策がある
- ユーザーエクスペリエンスを大きく向上させる
- 実装しないと競争力が低下する

**実践例**:

```
ECサイトのShould have:
✓ レビュー機能
✓ お気に入り機能
✓ 複数の決済手段
✓ 詳細な検索フィルター
✓ 注文履歴
```

### Could have（望ましい）

**定義**: あれば便利だが、なくても問題ない要件

**判断基準**:

- ビジネス価値が中程度（2点以下）
- 一部のユーザーのみが利用
- 実装コストが高い
- リソースに余裕があれば実装

**実践例**:

```
ECサイトのCould have:
✓ ソーシャルシェア機能
✓ ウィッシュリスト
✓ ギフトラッピング
✓ クーポン機能
✓ ポイント制度
```

### Won't have（対象外）

**定義**: 今回のスコープには含めない要件

**判断基準**:

- ビジネス価値が低い（1点以下）
- 実現可能性が低い（2点以下）
- リスクが高い（4点以上）
- 戦略的に優先度が低い

**実践例**:

```
ECサイトのWon't have:
✓ AR試着機能（技術的制約）
✓ サブスクリプション（別プロジェクト）
✓ マーケットプレイス機能（複雑度高）
✓ 多言語対応（国内のみ）
```

## スコアリングとMoSCoWの関係

```
優先度スコア計算式:
(ビジネス価値 × 3) + (実現可能性 × 2) - (リスク × 2) - (コスト × 1)

分類マッピング:
15-22点 → Must have
8-14点  → Should have
1-7点   → Could have
0点以下 → Won't have
```

## バランスガイドライン

```
理想的な分布:
Must have:   30-40%
Should have: 30-40%
Could have:  20-30%
Won't have:  明示的に記録

警告サイン:
❌ Must haveが50%超 → スコープが広すぎる
❌ Should haveが10%未満 → 柔軟性がない
❌ Could haveが40%超 → 優先順位が曖昧
```

## まとめ

MoSCoW分類により、明確なスコープ定義とMVP策定が可能になります。
