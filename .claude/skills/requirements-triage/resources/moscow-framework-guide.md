# MoSCoW分類フレームワーク詳細ガイド

## 概要

MoSCoW分類は、要件の優先順位を4つのカテゴリーに分類する手法です:
- **M**ust have（必須）
- **S**hould have（重要）
- **C**ould have（望ましい）
- **W**on't have（対象外）

このガイドでは、各カテゴリーの詳細な判断基準と実践的な適用方法を提供します。

---

## Must have（必須）

### 定義

プロジェクトの成功に絶対に必要な要件。これがなければプロジェクトは成立しません。

### 判断基準

```markdown
## 必須要件の5つの判断基準（いずれか1つでも該当すればMust have）

1. **コア機能**: システムの核心的な機能
   - この機能なしではシステムの目的を達成できない
   - プロジェクトの存在理由そのもの

2. **法的義務**: 法律、規制、コンプライアンス要件
   - GDPR、個人情報保護法等の遵守
   - 業界標準（PCI DSS等）の準拠

3. **契約上の義務**: 顧客との合意事項
   - SLA（Service Level Agreement）で約束した機能
   - 契約書に明記されている要件

4. **ビジネスクリティカル**: ビジネスの継続性に直結
   - この機能がないと収益が発生しない
   - 顧客が使用を開始できない

5. **依存性の基盤**: 他の要件の前提条件
   - 他の多くの機能がこの機能に依存
   - この機能がないと他の機能が実装できない
```

### 具体例

#### 例1: ECサイトの場合

```markdown
## Must have要件

### MH-001: 商品検索機能
- **理由**: コア機能
- **根拠**: 商品を見つけられなければ購入できない
- **影響**: この機能がないとサイトの目的を達成できない

### MH-002: カート機能
- **理由**: コア機能
- **根拠**: 商品をカートに入れられなければ購入できない
- **影響**: ECサイトの基本機能

### MH-003: 決済機能
- **理由**: ビジネスクリティカル
- **根拠**: 決済できなければ収益が発生しない
- **影響**: ビジネスモデルの根幹

### MH-004: 個人情報の暗号化
- **理由**: 法的義務
- **根拠**: 個人情報保護法の遵守
- **影響**: 法的リスク、信頼性の喪失

### MH-005: 在庫管理機能
- **理由**: 依存性の基盤
- **根拠**: 決済、配送等の機能がこれに依存
- **影響**: 他の機能が実装できない
```

#### 例2: SaaSダッシュボードの場合

```markdown
## Must have要件

### MH-010: ユーザー認証
- **理由**: 依存性の基盤
- **根拠**: すべての機能が認証に依存
- **影響**: 他の機能を実装できない

### MH-011: ダッシュボード表示
- **理由**: コア機能
- **根拠**: データ可視化がサービスの目的
- **影響**: サービスの存在理由

### MH-012: データエクスポート
- **理由**: 契約上の義務
- **根拠**: SLAで「データのエクスポート機能」を約束
- **影響**: 契約違反
```

### Must haveの制約

```markdown
## Must haveの適切な割合

**推奨割合**: 全要件の30-40%

**40%を超える場合の問題**:
- スコープが肥大化している
- MVP（Minimum Viable Product）の定義が曖昧
- 優先順位付けが機能していない

**30%未満の場合の問題**:
- 過度に厳格な判断基準
- プロジェクトの成功定義が不明確

**調整方法**:
1. Must haveの基準を再確認
2. Should haveとの境界を見直し
3. MVPの定義を明確化
4. ステークホルダーと再協議
```

## Should have（重要）

### 定義

ビジネス価値は高いが、短期的には回避策がある要件。
ユーザーエクスペリエンスを大きく向上させるが、必須ではない機能。

### 判断基準

```markdown
## Should haveの4つの判断基準（いずれか1つでも該当）

1. **高いビジネス価値**: ユーザー満足度、効率性の向上
   - ユーザー体験を大幅に改善
   - 業務効率を向上させる

2. **回避策が存在**: 一時的には手動で対応可能
   - この機能がなくてもサービスは提供できる
   - 手作業、代替手段で対処可能

3. **競争優位性**: 競合との差別化要素
   - あれば他社との差別化になる
   - 市場でのポジショニングに貢献

4. **将来の拡張性**: 長期的な成長の基盤
   - スケーラビリティの向上
   - 将来の機能追加の土台
```

### 具体例

#### 例1: ECサイトの場合

```markdown
## Should have要件

### SH-001: レビュー機能
- **理由**: 高いビジネス価値
- **根拠**: ユーザーの購買意思決定を支援、コンバージョン率向上
- **回避策**: 最初は管理者がレビューを手動で追加
- **影響**: ユーザー体験の向上、信頼性の向上

### SH-002: お気に入り機能
- **理由**: ユーザー満足度向上
- **根拠**: リピート購入を促進
- **回避策**: ユーザーがブックマークで対応
- **影響**: UXの向上、リピート率の向上

### SH-003: 商品レコメンデーション
- **理由**: 競争優位性
- **根拠**: パーソナライズされた体験の提供
- **回避策**: 人気商品を手動でキュレーション
- **影響**: 売上向上、ユーザーエンゲージメント向上

### SH-004: モバイルアプリ
- **理由**: 将来の拡張性
- **根拠**: モバイルユーザーの増加に対応
- **回避策**: レスポンシブWebサイトで対応
- **影響**: モバイルユーザーの利便性向上
```

#### 例2: SaaSダッシュボードの場合

```markdown
## Should have要件

### SH-010: カスタムレポート
- **理由**: 高いビジネス価値
- **根拠**: 顧客の多様なニーズに対応
- **回避策**: 標準レポートのみ提供、カスタマイズはサポート経由
- **影響**: 顧客満足度向上、解約率低下

### SH-011: リアルタイム通知
- **理由**: ユーザー満足度向上
- **根拠**: 迅速な対応が可能
- **回避策**: メールでの通知（遅延あり）
- **影響**: ユーザー体験の向上

### SH-012: API連携
- **理由**: 将来の拡張性
- **根拠**: サードパーティツールとの統合
- **回避策**: 手動でデータをエクスポート/インポート
- **影響**: ワークフローの自動化
```

### Should haveの実装戦略

```markdown
## フェーズ分けの戦略

### フェーズ1: MVPリリース（Must haveのみ）
- 期間: 3ヶ月
- スコープ: Must have要件のみ
- 目標: 市場投入、初期フィードバック収集

### フェーズ2: 価値向上リリース（Should have追加）
- 期間: 2ヶ月
- スコープ: 高優先度のShould have（2-3個）
- 目標: ユーザー体験の向上、競争力強化

### フェーズ3: 継続的改善（残りのShould have）
- 期間: 継続的
- スコープ: 残りのShould have要件
- 目標: プロダクトマーケットフィット、成長

**優先順位の判断基準**:
1. ユーザーフィードバックで最も要望が多い
2. 実装コストが低い（効果/コスト比が高い）
3. 技術的な依存関係（先に実装すべきもの）
```

## Could have（望ましい）

### 定義

あれば便利だが、なくても問題ない要件。
リソースに余裕があれば実装する。将来のバージョンで実装を検討する。

### 判断基準

```markdown
## Could haveの3つの判断基準（すべて該当）

1. **限定的な価値**: 一部のユーザーのみが利用
   - ニッチなユースケース
   - 使用頻度が低い

2. **高い実装コスト**: 工数、技術的難易度が高い
   - 開発に長期間を要する
   - 複雑な技術、新しい技術の習得が必要

3. **低い緊急性**: 今すぐ必要ではない
   - 市場からのプレッシャーがない
   - 競合も実装していない
```

### 具体例

#### 例1: ECサイトの場合

```markdown
## Could have要件

### CH-001: ダークモード
- **理由**: 限定的な価値
- **根拠**: 一部のユーザーのみが使用
- **コスト**: 中（UIの全体的な見直しが必要）
- **緊急性**: 低（今すぐ必要ではない）
- **影響**: 一部のユーザーの満足度向上

### CH-002: 音声検索
- **理由**: 高い実装コスト
- **根拠**: 音声認識APIの統合、UIの設計が必要
- **価値**: 限定的（モバイルユーザーの一部のみ）
- **緊急性**: 低（競合も未実装）
- **影響**: 先進的なUXの提供

### CH-003: AR（拡張現実）で商品プレビュー
- **理由**: 高い実装コスト + 低い緊急性
- **根拠**: AR技術の習得、3Dモデルの作成が必要
- **価値**: 限定的（対応デバイスが限られる）
- **緊急性**: 低（ニッチな機能）
- **影響**: 差別化、話題性

### CH-004: 多言語対応（マイナー言語）
- **理由**: 限定的な価値
- **根拠**: 主要言語（日本語、英語）以外の需要が不明
- **コスト**: 中（翻訳、テストが必要）
- **緊急性**: 低（グローバル展開は将来的）
- **影響**: グローバル展開の準備
```

### Could haveの扱い方

```markdown
## バックログでの管理

### 配置
- バックログの下位（優先度が低い位置）
- 「将来の検討項目」として分類

### 定期的な見直し
- 四半期ごとにレビュー
- 市場の変化、技術の進化を考慮
- Could have → Should haveへの昇格判断

**昇格の判断基準**:
1. ユーザーからの要望が増加
2. 技術的に実装が容易になった（新ライブラリ等）
3. 競合が実装し、競争上必要になった
4. 関連機能の実装により、実装コストが下がった

### 実装の判断
- リソースに余裕がある場合のみ
- スプリント計画で「余裕枠」として検討
- Must have、Should haveが完了してから
```

## Won't have（対象外）

### 定義

今回のスコープには含めない要件。
将来のバージョンで再検討する可能性があるもの、または完全に除外するもの。

### 判断基準

```markdown
## Won't haveの5つの判断基準（いずれか1つでも該当）

1. **スコープ外**: プロジェクトの目的と合致しない
   - ビジネス目標と無関係
   - ターゲットユーザーのニーズではない

2. **実現不可能**: 技術的、リソース的に不可能
   - 現在の技術では実装できない
   - 予算、人員が不足

3. **高リスク**: 失敗時の影響が極めて大きい
   - ビジネスリスクが高い
   - 技術的な不確実性が高い

4. **戦略的な判断**: 意図的に実装しない
   - 競合との差別化のため
   - ビジネスモデルと矛盾

5. **法的・倫理的問題**: 法律、倫理に反する
   - 規制に抵触
   - 倫理的に問題がある
```

### 具体例

#### 例1: ECサイトの場合

```markdown
## Won't have要件

### WH-001: 暗号通貨決済
- **理由**: 戦略的な判断 + 高リスク
- **根拠**: ビジネスモデル（法定通貨決済）と矛盾、法的リスク
- **将来の再検討**: 規制が整備され、ビジネス戦略が変わったら検討
- **代替案**: クレジットカード、銀行振込のみ

### WH-002: 自動翻訳（全商品説明）
- **理由**: 実現不可能（コスト）
- **根拠**: 数万点の商品説明を翻訳するコストが高すぎる
- **将来の再検討**: AI翻訳の品質が向上したら検討
- **代替案**: 主要商品のみ手動で翻訳

### WH-003: ブロックチェーン在庫管理
- **理由**: スコープ外 + 高リスク
- **根拠**: オーバーエンジニアリング、ビジネス価値が不明確
- **将来の再検討**: 再検討の予定なし
- **代替案**: 従来のデータベースで十分

### WH-004: ゲーミフィケーション機能
- **理由**: スコープ外
- **根拠**: ターゲットユーザー（ビジネス向け）のニーズではない
- **将来の再検討**: ターゲット市場が変わったら検討
- **代替案**: なし
```

### Won't haveの記録と管理

```markdown
## 記録の重要性

**なぜ記録するか**:
1. 同じ要望が繰り返し出てくるのを防ぐ
2. 判断理由を後から参照できる
3. 将来の再検討時に過去の議論を活用

**記録すべき項目**:
- 要件の説明
- Won't haveと判断した理由
- 判断日、判断者
- 将来の再検討の可能性（有/無）
- 代替案

**記録場所**:
- バックログツール（Jira等）の専用ステータス
- ドキュメント（意思決定ログ）
```

## MoSCoW分類のワークフロー

### ステップ1: 要件の収集

```markdown
## 収集方法
1. ステークホルダーインタビュー
2. ユーザーフィードバック
3. 市場調査、競合分析
4. 技術的な制約の洗い出し

## 収集した要件の一覧化
- 要件ID、要件名、説明を記録
- まだ優先順位は付けない
```

### ステップ2: Must haveの特定

```markdown
## プロセス
1. 5つの判断基準で評価
2. 「これがなければプロジェクトは成立しないか？」を問う
3. ステークホルダーと合意

## 注意点
- Must haveを多くしすぎない（30-40%が目安）
- 「あると良い」ではなく「絶対に必要」のみ
```

### ステップ3: Should haveとCould haveの分類

```markdown
## Should haveの判断
- 高いビジネス価値 + 回避策がある
- ユーザー体験を大幅に向上させる
- 競争優位性に貢献

## Could haveの判断
- 限定的な価値 + 高コスト + 低緊急性
- リソースに余裕があれば実装

## 境界線の判断
Should haveとCould haveの境界は、「ビジネス価値」と「実装コスト」のバランス
- ビジネス価値が高く、コストが低い → Should have
- ビジネス価値が低く、コストが高い → Could have
```

### ステップ4: Won't haveの確定

```markdown
## プロセス
1. スコープ外、実現不可能、高リスク要件を特定
2. ステークホルダーに説明
3. 判断理由を記録

## コミュニケーション
- なぜWon't haveと判断したかを明確に説明
- 将来の再検討可能性を伝える
- 代替案があれば提示
```

### ステップ5: レビューと調整

```markdown
## レビューポイント
- Must haveが30-40%の範囲内か？
- Should haveとCould haveのバランスは適切か？
- ステークホルダー全員が合意しているか？

## 調整
- Must haveが多すぎる場合、Should haveへの移動を検討
- 依存関係を考慮した順序付け
```

## MoSCoW分類のベストプラクティス

### 1. タイムボックスの設定

```markdown
## Must haveのタイムボックス
- MVP（Minimum Viable Product）リリースまでの期限を設定
- 例: 3ヶ月でMust haveをすべて完了

## Should haveのタイムボックス
- フェーズ2リリースまでの期限を設定
- 例: MVPリリース後2ヶ月でShould haveを完了

**効果**:
- スコープクリープの防止
- リリース計画の明確化
```

### 2. ステークホルダーの巻き込み

```markdown
## 合意形成のプロセス
1. MoSCoW分類のワークショップを開催
2. 各要件について議論
3. 投票や議論で合意形成

## 定期的なレビュー
- 四半期ごとにMoSCoW分類を見直し
- 市場の変化、フィードバックを反映
```

### 3. 透明性の確保

```markdown
## 判断基準の公開
- MoSCoW分類の判断基準を文書化
- すべてのチームメンバーが理解できるようにする

## 判断理由の記録
- なぜその分類になったかを記録
- 後から参照できるようにする
```

## まとめ

MoSCoW分類は、要件の優先順位を明確にし、スコープを管理するための強力なツールです:

- **Must have**: 絶対に必要（30-40%）
- **Should have**: 重要だが回避策あり
- **Could have**: あれば便利
- **Won't have**: 今回は対象外

適切なMoSCoW分類により、MVPを早期にリリースし、ユーザーフィードバックを基に継続的に価値を提供できます。
