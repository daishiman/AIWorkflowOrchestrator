# Socialization（共同化）詳細ガイド

## 概要

Socialization（共同化）は、暗黙知を他者と共有するプロセスです。
経験の共有、観察、模倣を通じて、言語化されていない知識が伝達されます。
これはSECIサイクルの起点となる重要なフェーズです。

## Socializationの特性

### 暗黙知の伝達

**定義**: 直接的な経験を通じた知識の共有

**特徴**:
- 言葉による説明が最小限
- 観察と模倣が中心
- 実践を通じた学習
- 感覚的な理解

**例**:
- コードレビューでの対話
- ペアプログラミング
- 実装の観察
- 問題解決プロセスの共有

## Claude Codeエコシステムでの適用

### 暗黙知の源泉

#### 源泉1: コードレビューコメント

**収集対象**:
- 頻繁に指摘される内容
- 「なぜそうすべきか」の説明
- 代替案の提示
- 設計判断の根拠

**収集方法**:
```bash
# GitHubのPRコメントから抽出
# "LGTM"以外のコメントを分析
# 繰り返し出現するパターンを特定
```

**分析観点**:
- 指摘の頻度: 3回以上繰り返される内容
- 指摘の深さ: 表面的 vs 根本的
- 指摘の一貫性: 同じ原則に基づいているか

#### 源泉2: チーム内議論

**収集対象**:
- 設計の意思決定プロセス
- トレードオフの議論
- 問題解決のアプローチ
- ベストプラクティスの発見

**収集方法**:
```
Slack/Discord:
- #tech-discussion チャンネル
- 設計レビュースレッド
- トラブルシューティングの会話

Issue/Discussion:
- 設計提案
- RFC（Request for Comments）
- 技術的議論
```

**記録フォーマット**:
```markdown
## 議論記録: [トピック]
- **日付**: YYYY-MM-DD
- **参加者**: [メンバーリスト]
- **トピック**: [議論の主題]
- **背景**: [なぜ議論が必要だったか]
- **主要な論点**:
  1. [論点1]
  2. [論点2]
- **結論**: [最終的な決定]
- **根拠**: [決定の理由]
- **抽出される原則**: [汎用化可能な原則]
```

#### 源泉3: 実装パターンの観察

**収集対象**:
- 繰り返し使われるコードパターン
- 問題を回避するための工夫
- パフォーマンス最適化の技法
- エラーハンドリングのパターン

**抽出方法**:

**パターン分析**:


## 暗黙知収集のテクニック

### テクニック1: 頻度分析

**原則**: 繰り返し出現する要素に注目

**手順**:
1. **データ収集**: 一定期間のレビューコメント、議論を収集
2. **キーワード抽出**: 頻出するキーワードや概念を抽出
3. **頻度カウント**: 各キーワードの出現回数を集計
4. **閾値設定**: 3回以上出現を形式知化の候補とする

**ツール**:
```bash
# レビューコメントから頻出ワード抽出
grep -o "\b[A-Za-z]\{5,\}\b" reviews.txt | sort | uniq -c | sort -rn
```

### テクニック2: パターン認識

**原則**: 類似した事例をグループ化

**手順**:
1. **事例の収集**: 10件以上の具体例を集める
2. **共通要素の発見**: 事例間の共通パターンを特定
3. **パターンの命名**: 認識しやすい名前を付与
4. **適用条件の定義**: どのような時に使うかを明確化

**例**:
```
事例収集:
- "UserRepositoryにビジネスロジック混在" × 5件
- "ProductServiceにDB直接アクセス" × 4件
- "OrderControllerにバリデーションロジック" × 6件

共通パターン:
「責務の混在」- 本来の責務と異なるロジックが混在

形式知化:
原則: レイヤー分離とクリーンアーキテクチャ
```

### テクニック3: 観察記録

**原則**: 専門家の作業プロセスを詳細に記録

**観察項目**:
- どのような順序で作業するか
- どの時点でどのような判断をするか
- どのようなツールを使うか
- 何を確認してから次のステップに進むか

**記録フォーマット**:
```markdown
## 観察記録: [専門家名] - [タスク名]

### 作業フロー
1. [ステップ1]
   - 判断: [何を確認したか]
   - ツール: [使用したツール]
   - 所要時間: [X分]

2. [ステップ2]
   - ...

### 気づき
- [観察から得られた洞察1]
- [観察から得られた洞察2]

### 抽出可能な原則
- [原則1]
- [原則2]
```

## 経験の言語化

### 言語化の段階

**段階1: 感覚の認識**
```
暗黙知: "何となく違和感がある"

認識: 違和感の正体を特定する試み
- 何に対して？
- どのような種類の違和感？
- 過去の似た経験は？
```

**段階2: 要因の特定**
```
認識: "コードの複雑さに違和感"

特定: 具体的な要因
- ネストが深い（4階層）
- 関数が長い（100行）
- 条件分岐が多い（10箇所）
```

**段階3: 基準の定義**
```
要因: ネスト、長さ、分岐の多さ

基準:
- ネスト: 3階層以下
- 関数: 50行以下
- 分岐: 5箇所以下
```

**段階4: 原則の抽出**
```
基準の背景にある原則:
- 認知負荷の最小化
- 可読性の維持
- テスト容易性の確保

→ 原則: シンプルさの原則（KISS: Keep It Simple, Stupid）
```

## 対話による知識抽出

### ソクラテス式問答

**目的**: 質問を通じて暗黙知を顕在化させる

**質問パターン**:

**パターン1: 判断基準の明確化**
```
Q: "なぜこの設計を選んだのですか？"
A: "スケーラブルだから"
Q: "何を基準にスケーラブルと判断しましたか？"
A: "ユーザー数が増えても性能が劣化しない"
Q: "具体的にどのように確認しましたか？"
A: "負荷テストで10倍のユーザーでも応答時間が1.5倍以内"

→ 基準: 10倍の負荷で応答時間が1.5倍以内
```

**パターン2: 代替案の検討**
```
Q: "他の選択肢は検討しましたか？"
A: "はい、パターンBも検討した"
Q: "なぜパターンAを選びましたか？"
A: "実装コストが低く、十分な性能があるため"
Q: "具体的なトレードオフは？"
A: "パターンB: 高性能だが実装に2週間、パターンA: 十分な性能で実装に3日"

→ 意思決定基準: コストと性能のバランス
```

## ベストプラクティス

### 収集時
1. **即座に記録**: 気づいた時点で記録、後回しにしない
2. **コンテキスト保持**: どのような状況での知識かを明記
3. **生のまま記録**: 整理や編集は後で、まずは記録

### 分析時
1. **十分な事例**: 3件以上で一般化を検討
2. **パターン優先**: 個別事例より共通パターンに注目
3. **反例の確認**: 当てはまらないケースも記録

### 形式知化移行時
1. **段階的**: Socialization → Externalization → Combination
2. **検証**: 抽出した原則が事例を説明できるか確認
3. **フィードバック**: 元の専門家に確認
