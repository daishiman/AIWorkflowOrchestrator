# 知識キュレーションフレームワーク

## 概要

知識キュレーションは、収集・評価・統合・更新の4つのフェーズから成るプロセスです。
このフレームワークにより、質の高い知識を継続的に維持し、組織全体で効果的に活用できます。

## 4つのキュレーションフェーズ

### Phase 1: 収集 (Capture)

**目的**: 価値ある情報を見逃さず収集する

#### 情報源の分類

**一次情報源** (最も信頼性が高い):

- 公式ドキュメント
- 学術論文
- 書籍（専門家著）
- 技術仕様書

**二次情報源** (信頼性を評価して使用):

- ブログ記事
- チュートリアル
- カンファレンストーク
- Stack Overflow回答

**三次情報源** (検証が必要):

- SNS投稿
- 個人の経験談
- 非公式なフォーラム

#### 収集基準

**優先度: 高**

- [ ] 一次情報源である
- [ ] 著者の専門性が確認できる
- [ ] 最新の情報である（発行から1年以内）
- [ ] 他の信頼できる情報源でも言及されている

**優先度: 中**

- [ ] 二次情報源だが著者が明確
- [ ] 実践的な内容を含む
- [ ] コミュニティで広く参照されている

**優先度: 低**

- [ ] 三次情報源
- [ ] 著者不明または専門性不明
- [ ] 古い情報（発行から3年以上）

#### 収集方法

**手法1: 構造化された検索**

```
キーワード定義 → 複数情報源で検索 → フィルタリング → 記録
```

**手法2: セレンディピティ的発見**

```
日常の作業中の気づき → 即座に記録 → 後で評価
```

**手法3: 体系的レビュー**

```
特定トピックの網羅的調査 → 情報源リスト作成 → 順次レビュー
```

#### 収集ツール

**Claude Codeでの実装**:

- Read: ドキュメント読み込み
- Grep: パターン抽出
- Web検索: 最新情報収集

**記録フォーマット**:

```markdown
## [情報タイトル]

- **情報源**: [URL/書籍名]
- **種別**: [一次/二次/三次]
- **信頼性**: [高/中/低]
- **日付**: YYYY-MM-DD
- **要約**: [2-3文で要点]
- **関連トピック**: [タグ]
```

### Phase 2: 評価 (Evaluate)

**目的**: 収集した情報の品質と適用可能性を判断する

#### 評価基準

**品質評価の3軸**:

**軸1: 正確性 (Accuracy)**

- [ ] 事実に基づいているか？
- [ ] 検証可能な根拠があるか？
- [ ] 他の信頼できる情報源と一致するか？
- [ ] 誤りや矛盾がないか？

**軸2: 完全性 (Completeness)**

- [ ] 必要な情報がすべて含まれているか？
- [ ] 前提条件が明示されているか？
- [ ] エッジケースや例外が説明されているか？
- [ ] 実践に必要な詳細が揃っているか？

**軸3: 明確性 (Clarity)**

- [ ] 理解しやすい言葉で書かれているか？
- [ ] 専門用語に説明があるか？
- [ ] 構造が論理的か？
- [ ] 例が適切に含まれているか？

#### 適用可能性の評価

**コンテキスト適合性**:

- [ ] Claude Codeエコシステムに適用可能か？
- [ ] プロジェクトの技術スタックと一致するか？
- [ ] チームのスキルレベルに適しているか？
- [ ] 実装コストは妥当か？

**普遍性 vs 特殊性**:

**普遍的な知識** (高い再利用性):

- 設計原則（SOLID、DRY等）
- アーキテクチャパターン
- 問題解決手法

**特殊な知識** (低い再利用性):

- プロジェクト固有の実装
- 特定バージョンの回避策
- 一時的な制約への対応

**判断**: 普遍的な知識を優先、特殊な知識は明確に区別して記録

#### 評価スコアリング

**総合評価スコア** (0-10点):

```
総合スコア = (正確性 × 0.4) + (完全性 × 0.3) + (明確性 × 0.3)

採用基準:
- 8-10点: 即座に採用
- 6-7点: 補完情報を追加して採用
- 4-5点: 再評価または情報源追加
- 0-3点: 不採用
```

#### 評価記録

```markdown
## 評価結果: [情報タイトル]

- **正確性**: [0-10点] - [理由]
- **完全性**: [0-10点] - [理由]
- **明確性**: [0-10点] - [理由]
- **総合**: [0-10点]
- **判定**: [採用/要補完/不採用]
- **アクション**: [次のステップ]
```

### Phase 3: 統合 (Integrate)

**目的**: 評価された情報を既存の知識体系に組み込む

#### 統合パターン

**パターン1: 追加型統合**

```
既存知識: A、B
新規知識: C
結果: A、B、C（独立した要素として追加）
```

**適用条件**: 新規知識が既存知識と独立している

**パターン2: 強化型統合**

```
既存知識: Aの概要
新規知識: Aの詳細
結果: Aの概要 + 詳細（既存知識を強化）
```

**適用条件**: 新規知識が既存知識を補完・詳細化する

**パターン3: 置換型統合**

```
既存知識: A（古い情報）
新規知識: A'（更新された情報）
結果: A'（既存知識を置換）
```

**適用条件**: 新規知識が既存知識を更新・修正する

**パターン4: 統合型**

```
既存知識: A、B
新規知識: C（AとBを統合する視点）
結果: C（A、B統合） + A参考 + B参考
```

**適用条件**: 新規知識が既存知識を統合する枠組みを提供

#### 重複の排除

**重複検出の方法**:

1. **キーワード検索**: Grepで既存知識を検索
2. **概念的重複**: 同じ原則を異なる言葉で表現
3. **情報源の重複**: 同じ情報源からの複数引用

**重複解消の戦略**:

- **マージ**: 2つの知識を1つに統合
- **参照**: 一方を主とし、他方は参照として残す
- **削除**: 明らかに劣る方を削除

#### 関連性の明確化

**知識間の関係タイプ**:

- **前提**: この知識を理解するために必要な知識
- **応用**: この知識を応用した発展的な知識
- **対比**: この知識と比較すべき代替的な知識
- **補完**: この知識と組み合わせて使う知識

**関連性の記述**:

```markdown
## 関連知識

- **前提**: [knowledge-A] - [関係性の説明]
- **応用**: [knowledge-B] - [関係性の説明]
- **対比**: [knowledge-C] - [関係性の説明]
```

#### 統合チェックリスト

- [ ] 既存知識との重複を確認し解消した
- [ ] 新規知識の位置づけ（独立/強化/置換/統合）を決定した
- [ ] 関連する知識への参照を追加した
- [ ] 知識の階層構造が論理的である
- [ ] Progressive Disclosure原則に準拠している

### Phase 4: 更新 (Update)

**目的**: 知識の鮮度を維持し、陳腐化を防ぐ

#### 更新トリガー

**トリガー1: 時間ベース**

- 最終更新から6ヶ月経過（定期レビュー）
- プロジェクトのメジャーバージョンアップ時
- 四半期末の一斉レビュー

**トリガー2: イベントベース**

- 参照する技術のバージョンアップ
- 新しいベストプラクティスの出現
- セキュリティ脆弱性の発見
- 非推奨APIの発表

**トリガー3: フィードバックベース**

- エージェントからの改善提案
- 使用時の問題報告
- 参照頻度の急激な低下
- 新しいユースケースの発見

#### 更新プロセス

**ステップ1: 陳腐化評価**

```
[ ] 技術バージョンは最新か？
[ ] 参照するAPIは有効か？
[ ] ベストプラクティスは現在も推奨されているか？
[ ] 実践で問題は発生していないか？
```

**ステップ2: 更新内容の決定**

- **Minor更新**: 軽微な修正、補足情報の追加
- **Major更新**: 大幅な構造変更、原則の変更

**ステップ3: バージョニング**

```
セマンティックバージョニング: major.minor.patch

major: 破壊的変更（既存の使い方が通用しなくなる）
minor: 機能追加（既存の使い方は引き続き有効）
patch: バグ修正、タイポ修正
```

**ステップ4: 変更履歴の記録**

```markdown
| バージョン | 日付       | 変更内容                                             |
| ---------- | ---------- | ---------------------------------------------------- |
| 2.0.0      | 2025-11-23 | メジャーアップデート: SECIモデルの適用手順を全面改訂 |
| 1.1.0      | 2025-09-15 | マイナーアップデート: トラブルシューティング追加     |
| 1.0.1      | 2025-08-10 | パッチ: タイポ修正                                   |
| 1.0.0      | 2025-08-01 | 初版作成                                             |
```

**ステップ5: 後方互換性の確認**

- [ ] 既存のエージェントが引き続き機能するか？
- [ ] 既存の参照が壊れていないか？
- [ ] 移行ガイドが必要か？

#### 更新サイクルの設計

**推奨レビューサイクル**:

- **月次**: 参照頻度の高い知識（Top 10%）
- **四半期**: 一般的な知識（Middle 80%）
- **半年**: 参照頻度の低い知識（Bottom 10%）

**自動化の可能性**:

- スクリプトによる陳腐化検出
- 参照頻度のモニタリング
- バージョン情報の自動チェック

## 品質メトリクス

### キュレーション効率

**指標1: 採用率**

```
採用率 = (採用された情報数 / 収集した情報数) × 100

目標: >70%
```

**指標2: 統合時間**

```
平均統合時間 = 総統合時間 / 統合した情報数

目標: <30分/件
```

### 知識品質

**指標3: 総合品質スコア**

```
平均品質スコア = Σ(各知識のスコア) / 知識総数

目標: >8.0点
```

**指標4: 陳腐化率**

```
陳腐化率 = (陳腐化した知識数 / 総知識数) × 100

目標: <10%
```

### 再利用性

**指標5: 参照率**

```
参照率 = (参照された知識数 / 総知識数) × 100

目標: >70%
```

## ベストプラクティス

### 収集時

1. **情報源を明記**: 必ず出典を記録
2. **即座に記録**: 後回しにせず、気づいた時点で記録
3. **タグ付け**: 検索しやすいようにタグを付与

### 評価時

1. **複数の視点**: 正確性・完全性・明確性の3軸で評価
2. **主観と客観**: 客観的基準を優先、主観は補助的に
3. **文書化**: 評価理由を記録し、後で検証可能に

### 統合時

1. **重複チェック**: 統合前に必ず既存知識を検索
2. **関連性の明示**: 他の知識との関係を明確に
3. **段階的統合**: 一度に大量ではなく、段階的に

### 更新時

1. **定期レビュー**: 6ヶ月ごとの確認サイクル
2. **バージョニング**: セマンティックバージョニングを遵守
3. **変更履歴**: すべての変更を記録

## トラブルシューティング

### 問題1: 情報過多で評価が追いつかない

**症状**: 収集した情報が溜まり、評価されずに放置

**解決策**:

1. 収集時に簡易評価（3段階）を実施
2. 優先度の高い情報から順次評価
3. 自動フィルタリングの導入

### 問題2: 重複が多く発生する

**症状**: 同じような知識が複数存在

**解決策**:

1. 統合前の検索を徹底
2. タグ付けルールの統一
3. 定期的な重複チェック

### 問題3: 陳腐化の検出が遅れる

**症状**: 古い情報が参照され続ける

**解決策**:

1. 自動陳腐化検出スクリプトの導入
2. 定期レビューサイクルの厳守
3. フィードバックループの確立
