# 意味のある命名

## 概要

良い命名は、コードの可読性において最も重要な要素の一つです。
適切な名前は、コードの意図を明確にし、コメントの必要性を減らします。

## 命名の原則

### 1. 意図を明確にする

名前は「なぜ存在するか」「何をするか」「どう使うか」を伝えるべきです。

**判断基準**:

- 名前だけで何をするか分かるか？
- コメントなしで理解できるか？
- コンテキストがなくても意味が通じるか？

### 2. 発音しやすい名前を使う

プログラミングは共同作業であり、コードについて議論できる必要があります。

**避けるべき例**:

- `genymdhms` → `generationTimestamp`
- `modymdhms` → `modificationTimestamp`

### 3. 検索しやすい名前を使う

単一の文字や数値定数は検索が困難です。

**避けるべき例**:

- `e` → `event` または `error`
- `7` → `DAYS_PER_WEEK`

## 品詞別命名規則

### 変数（名詞）

変数名は内容を表す名詞または名詞句を使用します。

**パターン**:

- 単数形: 単一の値 (`user`, `order`)
- 複数形: コレクション (`users`, `orderItems`)
- 形容詞+名詞: 特定の状態 (`activeUser`, `completedOrders`)

### 関数/メソッド（動詞）

関数名は動詞または動詞句で始まります。

**一般的な動詞**:

- `get`: 値を取得
- `set`: 値を設定
- `calculate`: 計算
- `validate`: 検証
- `create`: 生成
- `update`: 更新
- `delete`: 削除
- `find`: 検索
- `is/has/can`: ブール値を返す

### ブール値（is/has/can）

ブール値を返す関数や変数は、is/has/can/shouldで始めます。

**例**:

- `isValid`, `isActive`, `isCompleted`
- `hasPermission`, `hasChildren`
- `canExecute`, `canDelete`
- `shouldRetry`, `shouldNotify`

### クラス（名詞）

クラス名は名詞または名詞句を使用し、動詞は避けます。

**良い例**: `User`, `OrderProcessor`, `PaymentGateway`
**悪い例**: `Process`, `Handle`, `Execute`

## 避けるべき命名

### 曖昧な名前

**避けるべき例**:

- `data`, `info`, `temp`, `value`, `result`
- `doSomething`, `process`, `handle`

**対処**: より具体的な名前を使用

### 短すぎる名前

**避けるべき例**:

- `d`, `n`, `t` (ループ変数の`i`は例外)

**対処**: 意味が伝わる最小限の長さを使用

### 長すぎる名前

**避けるべき例**:

- `theUserThatHasTheCorrectPermissionToAccessThisResource`

**対処**: 冗長な情報を削除、コンテキストを活用

### エンコーディング

**避けるべき例**:

- ハンガリアン記法: `strName`, `intCount`
- 接頭辞/接尾辞: `m_name`, `_name`, `name_`

**対処**: 型情報は型システムに任せる

## コンテキストの活用

### クラスコンテキスト

クラス内では、クラス名がコンテキストを提供します。

**クラス外**: `userAddress.street`
**クラス内**: `street`（Addressクラス内）

### スコープの長さ

スコープが狭いほど短い名前が許容されます。

**ループ変数**: `i`, `j`, `k`（限定的スコープ）
**グローバル/クラス変数**: より説明的な名前
