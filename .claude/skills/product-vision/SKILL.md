# プロダクトビジョンスキル

## 概要

製品の長期的な方向性を定義し、測定可能な目標を設定することで、
チーム全体が同じ方向を向いて価値を創造できる環境を構築します。

## いつ使うか

- 新規プロダクトの立ち上げ
- 既存プロダクトの方向転換
- 年次/四半期計画の策定
- ステークホルダーへのビジョン共有
- チームのモチベーション向上

## コア概念

### プロダクトビジョンの要素

#### ビジョンステートメント
```
構成要素:
- Target: 誰のための製品か
- Need: どんな課題を解決するか
- Product: 製品カテゴリー
- Key Benefit: 主要な価値提案
- Differentiator: 差別化要因
```

#### エレベーターピッチテンプレート
```
[ターゲット顧客]向けの
[ニーズや機会]を解決する
[プロダクト名]は
[プロダクトカテゴリー]です。
これは[主要な価値提案]を提供し、
[競合製品]と違って
[差別化のポイント]という特徴があります。
```

### OKR (Objectives and Key Results)

#### Objectiveの特性
- **定性的**: インスピレーションを与える
- **野心的**: 現状を大きく超える目標
- **期限付き**: 四半期または年次
- **記憶可能**: シンプルで覚えやすい

#### Key Resultsの特性
- **定量的**: 数値で測定可能
- **挑戦的**: 70%達成で成功
- **具体的**: 曖昧さがない
- **期限明確**: いつまでに達成するか

## ビジョンボード作成

### Roman Pichlerのプロダクトビジョンボード

```
┌─────────────────────────────────────────┐
│            VISION                       │
│   製品が実現する理想の未来              │
├─────────────┬─────────────┬─────────────┤
│  TARGET     │   NEEDS     │  PRODUCT    │
│  GROUP      │             │             │
│  誰が使うか │ 何を解決    │ 何を作るか  │
├─────────────┼─────────────┼─────────────┤
│  BUSINESS   │  VALUE      │  SUCCESS    │
│  GOALS      │             │  METRICS    │
│ ビジネス目標│ 提供価値    │ 成功指標    │
└─────────────┴─────────────┴─────────────┘
```

### ビジョンキャンバス

```
1. 目的 (Purpose)
   - なぜこの製品が存在するか

2. 画像 (Picture)
   - 成功時の具体的なイメージ

3. 価値 (Values)
   - 守るべき原則

4. 方法 (Methods)
   - どのように実現するか

5. 障害 (Obstacles)
   - 克服すべき課題

6. 測定 (Measures)
   - 成功の判断基準
```

## ロードマップ策定

### ロードマップの種類

#### テーマベースロードマップ
```
Q1: ユーザー体験の向上
Q2: パフォーマンス最適化
Q3: エンタープライズ機能
Q4: AI/ML統合
```

#### 目標ベースロードマップ
```
Q1: ユーザー獲得 10,000人
Q2: 収益化開始 $100K MRR
Q3: 市場拡大 3カ国展開
Q4: プラットフォーム化
```

#### Now-Next-Later
```
Now (1-3ヶ月):
- 現在開発中
- コミット済み

Next (3-6ヶ月):
- 計画中
- 優先度高

Later (6ヶ月以降):
- 検討中
- 方向性のみ
```

### ロードマップ作成プロセス

1. **ビジョンの確認**
   - 長期目標の再確認
   - ステークホルダーとの整合

2. **テーマの特定**
   - ユーザーリサーチ
   - 市場分析
   - 技術トレンド

3. **優先順位付け**
   - インパクト vs 努力
   - 依存関係の考慮
   - リスク評価

4. **タイムライン設定**
   - マイルストーン定義
   - バッファの確保
   - 依存関係の可視化

5. **コミュニケーション**
   - ステークホルダー共有
   - 定期的な更新
   - 変更の透明化

## 戦略フレームワーク

### プロダクト戦略キャンバス

```
Vision → Strategy → Roadmap → Backlog

戦略の4P:
- Position: 市場でのポジション
- Plan: 実行計画
- Pattern: 行動パターン
- Perspective: 視点・価値観
```

### North Star Metric

```
選定基準:
- 顧客価値を反映
- 測定可能
- 行動可能
- 理解しやすい

例:
- Spotify: Time spent listening
- Airbnb: Nights booked
- Facebook: Daily active users
- Slack: Messages sent
```

### Value Proposition Canvas

```
顧客プロファイル:
- Customer Jobs: やりたいこと
- Pains: 困っていること
- Gains: 得たい成果

価値マップ:
- Products & Services: 製品・サービス
- Pain Relievers: 課題解決
- Gain Creators: 価値創造
```

## 成功指標の設定

### HEART Framework (Google)

```
H: Happiness (満足度)
  - NPS, CSAT, アンケート

E: Engagement (エンゲージメント)
  - DAU/MAU, セッション時間

A: Adoption (採用)
  - 新規ユーザー数, アクティベーション率

R: Retention (継続)
  - リテンション率, チャーン率

T: Task Success (タスク成功)
  - 完了率, エラー率, 所要時間
```

### Pirate Metrics (AARRR)

```
A: Acquisition (獲得)
  - トラフィック, 新規訪問者

A: Activation (活性化)
  - サインアップ, 初回利用

R: Retention (継続)
  - リピート率, 継続率

R: Revenue (収益)
  - LTV, ARPU, 売上

R: Referral (紹介)
  - 紹介率, バイラル係数
```

## ビジョンの浸透

### コミュニケーション戦略

#### 内部向け
```
- All Hands Meeting
- チームワークショップ
- ビジョンポスター
- Slackチャンネル
- 1on1での強化
```

#### 外部向け
```
- プレスリリース
- ブログポスト
- カンファレンス登壇
- ユーザーイベント
- ソーシャルメディア
```

### ストーリーテリング

```
構造:
1. Context: 背景と課題
2. Conflict: 対立や困難
3. Resolution: 解決策
4. Impact: インパクト
5. Call to Action: 行動喚起
```

## ビジョンの検証と調整

### 定期レビュー

```
月次:
- 進捗確認
- 障害の特定
- 軽微な調整

四半期:
- OKRレビュー
- ロードマップ調整
- ステークホルダー報告

年次:
- ビジョン見直し
- 戦略再評価
- 大規模な方向転換
```

### ピボットの判断基準

```
検討すべき兆候:
- 成長の停滞
- 市場の変化
- 競合の台頭
- 技術革新
- 規制変更

ピボットの種類:
- Zoom-in: 機能を製品化
- Zoom-out: 製品を機能化
- Customer: 顧客セグメント変更
- Platform: ビジネスモデル変更
- Technology: 技術基盤変更
```

## アンチパターンと対策

### ビジョンの形骸化
**症状**: 誰も覚えていない、参照されない
**対策**: 定期的な強化、日常での言及

### 短期思考の罠
**症状**: 四半期目標に振り回される
**対策**: 長期ビジョンとの整合性確認

### 机上の空論
**症状**: 現実と乖離したビジョン
**対策**: 定期的な市場検証、顧客フィードバック

## チェックリスト

### ビジョン作成時
- [ ] ターゲット顧客は明確か
- [ ] 解決する課題は重要か
- [ ] 差別化要因は明確か
- [ ] 測定可能な成功指標があるか
- [ ] チームが共感できるか

### ロードマップ作成時
- [ ] ビジョンと整合しているか
- [ ] 優先順位の根拠は明確か
- [ ] 依存関係は考慮されているか
- [ ] リソースは現実的か
- [ ] 変更の余地があるか
