## 1. SRP（単一責任の原則）

### 1.1 クラスレベル

- [ ] 各クラスの変更理由が1つに絞られている
- [ ] クラス名が単一の概念を明確に表現している
- [ ] クラスのメソッドがすべて同じ抽象度で動作している
- [ ] クラスの行数が300行以下である

### 1.2 モジュールレベル

- [ ] モジュールが単一の機能領域に特化している
- [ ] 異なるアクター（利用者グループ）向けの機能が分離されている
- [ ] importが一貫したレイヤーからのみ行われている

## 2. OCP（開放閉鎖の原則）

### 2.1 拡張性

- [ ] 新機能追加時に既存コードの修正が不要である
- [ ] 拡張ポイント（インターフェース/抽象クラス）が定義されている
- [ ] Strategyパターン等による振る舞いの差し替えが可能である

### 2.2 型による分岐

- [ ] switch文での型判定がない
- [ ] if-elseでのinstanceofチェックが最小限である
- [ ] typeフィールドによる分岐がない

## 3. LSP（リスコフの置換原則）

### 3.1 契約の遵守

- [ ] 派生クラスが基底クラスのすべてのメソッドを意味的に実装している
- [ ] 派生クラスが新たな例外を投げていない
- [ ] 事前条件を強化していない
- [ ] 事後条件を緩和していない

### 3.2 実装の完全性

- [ ] 空のメソッド実装（{}）がない
- [ ] NotImplemented例外を投げるメソッドがない
- [ ] クライアントコードで型による分岐がない

## 4. ISP（インターフェース分離の原則）

### 4.1 インターフェースサイズ

- [ ] インターフェースが10メソッド以下である
- [ ] 各インターフェースが単一の役割を表現している
- [ ] クライアントが使わないメソッドに依存していない

### 4.2 実装の完全性

- [ ] 実装クラスがすべてのメソッドを意味的に実装できている
- [ ] 空の実装やUnsupportedOperation例外がない
- [ ] 役割ベースのインターフェース分割が行われている

## 5. DIP（依存性逆転の原則）

### 5.1 依存の方向

- [ ] 上位モジュールが具体的な実装をimportしていない
- [ ] インターフェースが上位モジュール側で定義されている
- [ ] 依存性がコンストラクタで注入されている

### 5.2 具象への依存

- [ ] コンストラクタ内でnewを使用していない
- [ ] getInstance()パターンを使用していない
- [ ] フレームワーク固有のクラスに直接依存していない

### 5.3 テスタビリティ

- [ ] モックやスタブに差し替え可能である
- [ ] 外部サービスがインターフェースで抽象化されている
